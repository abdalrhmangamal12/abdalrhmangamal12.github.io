SafeSight: GPS-Denied Drone Inspection via LiDAR-Inertial Odometry

Modular perception device (mountable on drones, mobile robots, or handheld) that provides real-time LiDAR-inertial odometry to PX4 as external VIO — enabling position-hold and autonomous inspection in underground and industrial GPS-denied environments. Multiple releases shipping LIO-RF, FAST-LIO2, and Point-LIO backends.

LiDAR-Inertial SLAM GPS-Denied Navigation PX4 External VIO Ouster OS1-128 VectorNav VN-310 FAST-LIO2 Point-LIO LIO-RF Fast DDS + SHM ROS 2 Humble MAVLink EKF2
Problem

Position control where GPS cannot reach

Underground tunnels, mines, and industrial interiors have no GPS and unreliable barometers. Without an external position source, PX4's EKF2 falls into inertial dead-reckoning within seconds — uncertainty explodes, position-hold drifts, and the drone becomes uncontrollable. The SafeSight device solves this by supplying continuous, high-rate LiDAR-inertial odometry as the drone's primary position aiding source.

Architecture

LiDAR-inertial odometry → PX4 EKF2 pipeline

1 — Sensor Synchronisation & Calibration
LiDAR-to-IMU time synchronisation using hardware PPS and software timestamp alignment. Extrinsic calibration between Ouster OS1-128 and VectorNav VN-310. Handling of different reference frames and coordinate transforms (sensor → body → NED/ENU → PX4 local).
PPS sync → extrinsic cal → TF chain: sensor → body → PX4 local
2 — LiDAR-Inertial Odometry (Multi-Backend)
Release 1: LIO-RF — initial backend for proof-of-concept flights.
Release 2: FAST-LIO2 and Point-LIO — tightly-coupled iterated-EKF backends with improved accuracy and robustness in degenerate geometries. Each backend outputs 6-DOF pose at IMU rate.
OS1-128 + VN-310 → LIO-RF / FAST-LIO2 / Point-LIO → 6-DOF pose
3 — High-Throughput DDS Transport
Ouster OS1-128 at 20 Hz produces ~6–8 MB/scan. Configured Fast DDS with shared memory (SHM) transport to handle point cloud throughput without packet loss. Tuned maxMessageSize, fragment sizes, and kernel socket buffers for reliable intra-host delivery.
Fast DDS SHM → maxMessageSize tuning → kernel rmem_max
4 — PX4 External Vision Integration
Odometry pose streamed to PX4 via MAVLink ODOMETRY message as external vision input to EKF2. Configured EV aiding parameters, timestamp conventions, covariance scaling, and delay compensation. Validated fusion via estimator_aid_src_ev_* and vehicle_local_position diagnostics.
LIO pose → MAVLink ODOMETRY → EKF2 EV fusion → position-hold
5 — Failsafe & Robustness Hardening
Diagnosed and resolved position-hold instability when odometry drops: EKF2 falls to dead-reckoning (dead_reckoning=1), xy_valid drops, eph saturates. Implemented failsafe gating: position mode blocked unless local position is valid; automatic fallback to Altitude/Stabilised on aiding loss. Validated with controlled no-odometry isolation flights.
EKF health monitoring → mode gating → graceful degradation
Diagnostic Deep-Dive

EKF2 failure analysis

Conducted systematic flight-log analysis across a series of flights (good baseline + failed + controlled isolation) to diagnose position-hold instability. Identified the failure signature:

IndicatorHealthyFailure
dead_reckoning0 (aiding active)1 (inertial drift)
xy_valid10 (position invalid)
ephLow, stableSaturating / diverging
xy_reset_counterStableIncrementing repeatedly

Root causes investigated: odometry stream dropout, timestamp/timesync mismatches, covariance scaling issues, and insufficient update rate. Controlled isolation flights (odometry intentionally stopped) confirmed the failure signature matches "aiding missing."
